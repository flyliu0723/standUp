既然要梳理核心的业务逻辑，那我们就抛开那些花里胡哨的视觉，直接用**大厂产品架构**的思维，把这个软件的“底层核心状态机（State Machine）”给理清楚。

一个健康提醒软件最容易翻车的地方，就是**状态冲突**（比如：用户明明去开会了，回来发现软件判定他久坐了4个小时；或者中午午休时，软件疯狂弹窗）。

为了让倒计时、坐下、站起来、暂停这些动作能够完美闭环，我们需要定义**4个核心状态**和一套**状态流转逻辑**。

---

### 一、 核心状态定义

软件在任何时刻，必须且只能处于以下4个状态之一：

1. **坐下（Sitting / Focus）**：正在工作，久坐倒计时**递减**（如40分钟往0走），总坐姿时间**递增**。
2. **站立（Standing / Rest）**：正在休息，站立倒计时**递减**（如建议站立5分钟），总站立时间**递增**。
3. **暂停（Paused）**：用户手动暂停，或者午休时间，所有计时**全部冻结**。
4. **离座（Away / Auto-Detected）**：触发了空闲检测（键鼠无操作），所有计时**暂时冻结**，等待用户回来后进行“状态追溯”。

---

### 二、 核心动作与状态流转逻辑（核心看这里）

我们需要把用户的“显式操作（点按钮）”和“隐式行为（键鼠状态）”结合起来：

#### 1. 正常工作循环（坐下 -> 站立 -> 坐下）

* **【动作：开始上班 / 坐下】**：
* **逻辑**：重置“久坐倒计时”为设定的40分钟，开始**递减**。
* **记录**：在历史流水账中生成一条新记录，标记 `开始时间`，`状态=久坐中`。


* **【倒计时归零 -> 触发提醒】**：
* 界面进入强提醒或微提醒状态。


* **【动作：用户点击“我已起立”】**（或检测到离座）：
* **逻辑**：“久坐倒计时”停止；更新刚才那条流水账的 `结束时间`、`时长`，`结束原因=手动起立`。
* 同时，切换到“站立状态”：启动一个“站立目标倒计时”**（比如5分钟），开始**递减。总站立时间开始**递增**。


* **【站立倒计时归零 -> 提醒坐下】**：
* 提示用户：“休息结束，该回工位搬砖了”。用户点击“坐下”后，重新进入【正常工作循环】。



#### 2. “暂停”动作的介入

* **【动作：用户手动点击“暂停”】**（比如要线上开会、或者高能聚焦不希望被打扰）：
* **逻辑**：无论是久坐倒计时还是站立倒计时，**全部暂停**。
* **记录**：当前进行的流水账记录暂时挂起，不计入有效坐姿时间。


* **【动作：恢复】**：
* **逻辑**：从暂停前的状态和剩余时间继续往下走。



#### 3. “空闲检测”的智能追溯逻辑（最关键的体验闭环）

当用户起立时，他们往往**不会**去点“我已起立”。这时候“空闲检测”必须无缝接管：

* **场景 A：用户真的去接水/上厕所了（起立了，但没点按钮）**
* 当键鼠连续 5 分钟无操作，触发“离座状态”。
* **底层逻辑**：把当前时间**往前倒推 5 分钟**，作为“久坐结束时间”。
* **当用户回来动鼠标时**：弹出一个极简的二级确认（或者直接默认）：“你刚刚是去休息了吗？”。如果用户确认，这中间的时间直接计入“有效站立时间”，并自动重置下一个久坐循环。


* **场景 B：用户只是坐在位子上看文档/看视频（没动键鼠，但没起立）**
* 同样键鼠 5 分钟无操作，触发“离座”。
* **当用户回来动鼠标时**：如果用户选择“我没有离开”，系统自动把刚刚过去的空闲时间**重新追加回“总坐姿时间”**，久坐倒计时继续扣减。

### 三、 联动结算逻辑（报告的数据来源）

为了支撑我们之前规划的酷炫报告，每次状态切换都要在本地数据库（比如 Electron 常用的 Lowdb 或 SQLite）存下以下字段：

```json
{
  "id": "随机ID",
  "date": "2026-06-18",
  "type": "SITTING | STANDING", 
  "start_time": "16:20:00",
  "end_time": "17:00:00",
  "duration_seconds": 2400,
  "end_by": "MANUAL | AUTO_IDLE | TIMEOUT" 
}

```

* **总坐时** = 今天所有 `type: SITTING` 的 `duration_seconds` 总和。
* **最长连续久坐** = 今天所有 `type: SITTING` 记录中，`duration_seconds` 最大的那一组。
* **达标率** = `有效站立次数 / 触发提醒的总次数`。

大厂的产品逻辑讲究“宽进严出”：前端交互上给用户最少的操作负担（靠键鼠空闲检测去猜），但后台的数据流转必须黑白分明，每一个秒数都要有据可查。这样你做出来的软件不仅用起来聪明，生成的图表才会绝对精准。
